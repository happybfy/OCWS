# 门禁者验收规范

本文档定义门禁者（Gatekeeper）在阶段二「结果验收」中的验收标准和执行规范。核心原则：**验收方式必须匹配任务类型，避免形式主义验收（室内验收）。**

## 适用范围

本规范适用于：

- 门禁者（小智）对 `done/` 状态任务的验收操作
- 所有进入 `done/` 状态的任务，无论任务类型
- 验收报告写入任务 `logs/` 目录

## 核心原则

### 验收方式必须匹配任务类型

不同的任务类型需要不同的验收手段。禁止对所有任务使用统一的「看代码 + 跑一下」验收方式。

### 外部可见的行为必须外部验证

如果一个任务的目标是修复用户可在浏览器中观察到的 Bug，那么验收必须在浏览器中完成。代码审查、API 测试、日志检查都不能替代浏览器实测。

### 验收证据必须可追溯

所有验收结论必须附带可验证的证据（截图、curl 输出日志、浏览器 console 输出等），写入任务 `logs/` 目录。

---

## 任务类型分类与验收标准

### 类型 A：后端 / 代码任务

**定义**：任务目标不涉及用户界面变化，仅涉及后端逻辑、数据处理、API 接口、脚本、构建等。

**典型场景**：
- API 接口开发或修改
- 数据处理脚本
- 数据库变更
- 构建/部署配置
- 定时任务/后台服务
- 纯逻辑 Bug 修复（如计算错误、数据异常）

**验收标准（必须全部满足）**：

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | API curl 验证 | 对受影响接口执行 curl，确认状态码和响应体符合预期 |
| 2 | 日志检查 | 确认无错误/异常日志，关键操作有日志记录 |
| 3 | Build 通过 | 确认代码无编译/语法错误 |
| 4 | 边界情况 | 至少覆盖一个正常情况和两个边界/异常情况 |
| 5 | 回归检查 | 确认修改未破坏已有功能（至少抽查 1-2 个关联接口） |

**验收方式**：直接通过 SSH 在目标服务器上执行验证命令。

**验收通过条件**：以上 5 项全部通过。

---

### 类型 B：前端 / UI 任务

**定义**：任务目标涉及用户可直接观察到的界面变化、交互行为或视觉呈现。

**典型场景**：
- 页面布局修改
- CSS 样式调整
- JavaScript 交互逻辑
- 前端 Bug 修复（如显示异常、按钮不响应、表单验证问题等）
- 新增页面或组件
- 响应式适配

**验收标准（必须全部满足）**：

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 浏览器实测 | 在真实浏览器中打开目标页面，执行完整操作流程 |
| 2 | 抓屏证据 | 截取关键状态的页面截图，至少包含：操作前、操作中、操作后 |
| 3 | Console 检查 | 打开浏览器开发者工具 Console 面板，确认无 JavaScript 报错 |
| 4 | 交互验证 | 对页面执行用户操作（点击、输入、切换等），确认响应正确 |
| 5 | 多路径验证 | 至少覆盖正常路径和一个异常/边界路径 |

**验收方式**：
- 方式一（优先）：通过 Playwright/Puppeteer 等浏览器自动化工具执行验收
- 方式二（备选）：手动在浏览器打开并截图（用于无法自动化的场景）

**验收通过条件**：以上 5 项全部通过。**缺少浏览器实测证据（截图或自动化日志）的，一律不放行。**

---

### 类型 C：混合任务

**定义**：任务同时涉及前端和后端变更。

**验收标准**：
- 必须同时满足类型 A 和类型 B 的全部验收标准
- 后端部分按类型 A 验收，前端部分按类型 B 验收

---

### 类型 D：文档 / 配置任务

**定义**：任务仅涉及文档编写、配置修改（非代码逻辑变更）。

**验收标准**：

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 存在性检查 | 确认目标文件存在且内容非空 |
| 2 | 格式校验 | 检查 Markdown/YAML/JSON 格式是否正确 |
| 3 | 内容完整 | 确认关键字段/章节已填写 |

## 验收判定规则

### 判定结果

| 结果 | 含义 | 后续动作 |
|------|------|---------|
| ✅ 通过 | 全部验收标准满足 | task.md 状态 → `archived`，验收报告写入 `logs/`，通知计划者 |
| ❌ 退回 | 存在未满足的验收标准 | task.md 状态保持 `done`，验收报告列出问题，通知计划者决定下一步 |
| ⚠️ 有条件通过 | 存在不影响核心功能的小问题 | 记录问题，通知计划者决定是否先归档再追加修正任务 |

### 退回时必须包含

退回通知中必须明确列出：

1. 未满足的验收标准（引用本文档条款号）
2. 具体问题描述
3. 建议的修正方向

## 禁止事项（红线）

以下行为视为**验收违规**：

| # | 违规行为 | 示例 |
|---|---------|------|
| 1 | **室内验收** | 前端 Bug 任务仅通过读代码判断「逻辑正确」就放行，未在浏览器实测 |
| 2 | **降级验收** | 类型 B 任务按类型 A 标准验收（只跑 curl 不看页面） |
| 3 | **无证据验收** | 验收结论未附带任何可验证证据（截图/日志） |
| 4 | **越权执行** | 门禁者自行修改产出物后放行（只能退回，不能修） |
| 5 | **跳过验收** | 未执行验收就将任务标记为 `archived` |

### 室内验收的定义

> 「室内验收」指门禁者仅通过阅读代码、检查文件或在自己环境中跑一下——而非在目标运行环境中实际验证——就做出验收通过决定的验收方式。

**特别说明**：TASK-008 事件是典型的「室内验收」违规——前端 Bug 修复任务仅通过代码审查就被放行，未在浏览器中实测，导致 Bug 未被真正修复。

**处罚**：发现「室内验收」行为，视为验收流程事故，需通知计划者和项目发起人。

## 任务类型判断规则

### 门禁者如何判断任务类型

1. **首选**：查看 `task.md` 中执行者标注的任务类型字段（如有）
2. **备选**：根据以下特征自动判断：
   - 涉及 HTML/CSS/JS/前端文件 → 类型 B
   - 仅涉及后端代码/脚本/API → 类型 A
   - 涉及文档/Markdown/配置 → 类型 D
   - 同时涉及前后端 → 类型 C
3. **兜底**：无法判断时，按类型 C（混合）处理，两者都验

### task.md 建议格式

为便于门禁者判断，建议执行者在 `task.md` 中标注任务类型：

```markdown
## 任务信息

- **任务类型**：前端 / 后端 / 混合 / 文档
```

## 验收报告格式

验收报告写入 `{任务路径}/logs/verification-report-{日期}.md`，格式如下：

```markdown
# 验收报告

- **任务**：{task_id}
- **验收人**：{机器人 ID}
- **验收时间**：{ISO 8601}
- **任务类型**：{类型 A/B/C/D}
- **验收方式**：{浏览器实测 / API curl / 文件检查 / 混合}

## 验收项

| # | 检查项 | 结果 | 证据 |
|---|--------|------|------|
| 1 | ... | ✅/❌ | 见附件/截图路径 |
| ... |

## 判定

✅ 通过 / ❌ 退回 / ⚠️ 有条件通过

## 备注

{如有退回，列出具体问题和建议修正方向}
```

---

## 与已有规范的衔接

- 本规范是对 `task-lifecycle.md` 中任务状态流转规则的补充——明确 `done → archived` 这一环节的验收操作标准
- 验收报告必须写入 `task-lifecycle.md` 规定的任务 `logs/` 目录
- 验收通过后的归档操作，由门禁者通知计划者执行，遵循 `task-lifecycle.md` 的 `done → archived` 流转规则
- 退回后的重分配，遵循 `task-lifecycle.md` 的 `failed → assigned` 流转规则，由计划者处理
