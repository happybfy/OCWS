# 命名规范

本文档定义 OCWS 第一版的统一命名规范。所有项目、任务、通知、机器人和核心说明文件都必须遵守本规则，避免随着目录数量增加而出现检索困难、排序混乱和自动化解析不稳定的问题。

## 适用范围

本规范适用于以下对象：

- `10-projects/` 下的项目目录
- 项目下的任务目录
- `20-robots/` 下的机器人目录
- 机器人收件箱中的通知文件
- 项目和任务的核心说明文件

## 总体原则

1. 先唯一，再可读  
   命名首先要保证可唯一识别，其次才是可读性。

2. 先稳定，再美观  
   一旦目录或文件已经进入协作流程，非必要不应重命名。

3. 机器可解析  
   命名应尽量固定结构，便于脚本、机器人或检索工具解析。

4. 统一使用 ASCII 字符  
   标识符、目录名、文件名默认使用英文字母、数字和连字符 `-`，避免跨平台兼容性问题。

## 项目目录命名

项目目录必须使用以下格式：

```text
PRJ-<三位或更多数字编号>-<短名称>/
```

示例：

```text
PRJ-001-demo/
PRJ-002-order-assistant/
PRJ-015-content-pipeline/
```

要求如下：

- 前缀必须为 `PRJ-`
- 编号必须为顺序编号，建议至少三位，如 `001`
- 短名称必须使用小写英文和连字符
- 短名称应表达项目语义，不应过长

禁止示例：

```text
project-demo/
prj_001_demo/
PRJ-demo/
PRJ-001-订单系统/
```

## 任务目录命名

任务目录必须使用以下格式：

```text
TASK-<三位或更多数字编号>-<短名称>/
```

示例：

```text
TASK-001-research/
TASK-002-implement/
TASK-003-review/
```

要求如下：

- 前缀必须为 `TASK-`
- 编号在单个项目范围内递增
- 短名称需描述任务目的，不描述执行人
- 不允许把状态写入任务目录名

禁止示例：

```text
TASK-001-done/
task-001-review/
TASK-review/
```

## 通知文件命名

通知文件必须使用以下格式：

```text
NOTICE-<任务目录名>.md
```

示例：

```text
NOTICE-TASK-002-implement.md
NOTICE-TASK-015-review.md
```

要求如下：

- 通知文件名必须与目标任务目录名一一对应
- 通知文件必须保留 `.md` 扩展名
- 同一个机器人同一时刻不应存在多个指向同一任务的有效通知文件

## 机器人目录命名

机器人目录必须使用以下格式：

```text
robot-<短名称>/
```

示例：

```text
robot-writer/
robot-coder/
robot-reviewer/
```

要求如下：

- 前缀必须为 `robot-`
- 短名称使用小写英文和连字符
- 一个名称对应一种稳定职责，不建议频繁改名

## 核心说明文件命名

第一版优先统一以下文件名：

- 项目说明：`project.md`
- 任务说明：`task.md`
- 机器人说明：`profile.md`
- 目录说明：`README.md`

要求如下：

- 同类型对象使用相同文件名，避免不同项目各自命名
- 文件内容变化不应通过改文件名表达
- 状态应写入文件内容，不应写入文件名

## 编号规则

编号规则采用“局部唯一”策略：

- 项目编号在 `10-projects/` 范围内唯一
- 任务编号在单个项目范围内唯一
- 机器人不使用数字编号，使用稳定名称标识

当前版本不强制要求编号连续，但要求：

- 新编号不得复用已归档对象编号
- 同一层级不得出现重复编号
- 编号一旦对外引用，不应修改

## 重命名规则

以下情况下允许重命名：

- 对象尚未进入协作流程
- 命名存在明显错误且尚未被其他文档引用
- 已完成统一替换并确认无遗留引用

以下情况下不建议重命名：

- 已分发给机器人执行
- 已被外部文档或通知文件引用
- 已进入归档或历史记录

## 与自动化的关系

后续如引入脚本或自动派发能力，默认以本命名规范作为解析基础。因此，任何偏离本规范的目录和文件都应被视为非标准对象，需要人工处理。
