# OCWS 实战经验总结

> 基于 6 个真实项目的实战反馈。  
> 版本：v1.0 | 最后更新：2026-06-12

---

## 一、已验证有效的设计

### 1.1 目录即协议

共享文件系统 + 标准目录结构 + Markdown 文件，不需要 API、消息队列或数据库。

- ✅ 机器人跨主机读写，共享存储原子 `mv` 操作保证任务领取互斥
- ✅ 人类可直接 `ls` / `cat` 查看状态，无需额外工具
- ✅ 零运维成本：不需要维护中间件、不需要处理连接/认证/序列化

### 1.2 通知是指针，不是副本

通知文件只含任务 ID、路径、状态、分配对象、优先级，不复制任务正文。

- ✅ 避免信息分叉：task.md 永远是唯一事实源
- ✅ 通知小（几百字节），生成快、传输轻
- ✅ 冲突时以 task.md 为准，无需做差异合并

### 1.3 角色边界清晰

| 角色 | 职责 | 不越界 |
|------|------|--------|
| Planner | 读 inbox、拆任务、分发通知、重分配 failed | 不执行、不验收 |
| Executor | 原子领取、执行、产出、回写 task.md | 不规划、不验收 |
| Gatekeeper | 巡检 working/、验收 done/、决定归档/退回 | 不修改产出、不代执行 |
| Sponsor | 建项目、定任务、保障基础设施 | 不分发、不执行 |

### 1.4 轮询 + 事件触发双通道

- 各机器人有 cron 轮询（如每 5 分钟），保底不丢任务
- Sponsor 可通过 SSH + CLI 即时唤醒 Planner
- 两条通道互补：轮询保证最终一致性，CLI 触发降低延迟

## 二、实践中的关键教训

### 2.1 权限问题：共享存储 root_squash

共享存储目录下，文件属主为本地用户，远程机器的 root 用户被 squash 为 nobody。

**教训：**
- 通知文件和共享文档必须设 `chmod 644` 或更宽松权限
- 在 OCWS 目录中，统一组权限并保持组可写

### 2.2 文件即状态，但要防并发

`mv` 在共享存储上是原子操作，这是任务领取互斥的基础。

- 不要用 `cp` + `rm`（两步，非原子）
- 不要用编辑器直接修改已被其他人持有的文件
- Gatekeeper 验收时读 `output/`，不写

### 2.3 机器人断连恢复

机器宿主机关机重启后可能出现：
- 残留进程占用端口
- 进程管理器与独立进程冲突
- 通信代理间歇超时

**恢复 SOP：**
1. 确认机器已开机
2. SSH 登录，检查进程状态
3. 如端口冲突：`fuser -k <port>`，清理锁文件，重启
4. 如通信超时：等待代理恢复，或重启
5. 硬重启作为最后手段（清除所有残留状态）

### 2.4 项目重新激活

已完成（`completed`）的项目，如需追加任务：
1. 更新 `project.md`：状态 → `active`，补充本期范围和验收标准
2. 新增 `TASK-xxx/` 目录和 `task.md`
3. 更新受影响依赖链
4. 更新 `project.md` 变更记录
5. 通知 Planner 重新分发

### 2.5 一句「来活了」足够

Sponsor 通知 Planner 时不需要复述任务细节。

**正确做法：**
1. 更新 OCWS project.md / task.md（唯一事实源）
2. 在 Planner inbox 投递简短通知文件
3. 唤醒 Planner：「来活了」
4. Planner 自行读取 OCWS 文档并分发

**错误做法：**
- 在通知/消息中粘贴代码片段
- 口述执行步骤（与 task.md 可能不一致）
- 跳过 OCWS 文档直接指挥执行者

### 2.6 操作脚本的事务陷阱

在交互式控制台（如数据库 console、bench console）中执行操作脚本时，需注意：
- 部分交互式环境默认包装事务，`insert()` 后需显式 `commit()`
- 退出 console 时未提交的事务会回滚，导致操作"假成功"
- 验证操作结果时应**跨 session** 查询，不能依赖同一 session 内反馈

**教训：**
- 关键操作后立即 `commit()`
- 验证时必须新开 session，模拟"外部观察者"视角
- `task.md` 脚本中应内嵌 commit 调用，不依赖隐式行为

## 三、优化方向

### 3.1 通知标准化

当前有两种通知格式。建议统一：
- 任务级通知：`NOTICE-TASK-{编号}-{短名}.md`
- 项目级通知：`NOTICE-PRJ-{编号}-{意图}.md`

### 3.2 Sponsor 唤醒自动化

当前需要手动 SSH + CLI。后续可以：
- 封装唤醒脚本（`wake-planner.sh`）
- 缩短 Planner 的 cron 间隔（≤2 分钟）
- 或不依赖唤醒，仅靠短间隔轮询

### 3.3 基础设施自愈

考虑为 Gatekeeper 增加基础设施监控职责：
- 定期探测各机器人健康状态
- 发现异常时通知 Sponsor 或尝试自动恢复

### 3.4 共享存储故障降级

如果共享存储意外宕机：
- 所有 OCWS 操作暂停
- 需人工恢复
- 可考虑备选共享方案（如 rsync 定时同步）

### 3.5 扩展执行者

是否增加第三个执行者取决于实际负载：
- 观察指标：任务排队超 30 分钟、故障接管频繁触发
- 当前 2 执行者 + 短作业模式在 15+ 任务中表现稳定
- 增加前应先评估边际成本（维护新机器 + 通信配置 + 资源消耗）

---

## 四、总结

OCWS 的设计哲学验证有效：**简单 > 完备，约定 > 工具，文档 > 消息**。

多个项目下来，开销最大的不是写代码，而是建立协作秩序。一旦秩序建立，后续项目边际成本骤降。这是一次性投入、长期受益的设计。
