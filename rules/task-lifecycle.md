# 任务状态流转规范

本文档定义 OCWS 中任务从创建到归档的标准状态流转规则。目标是让项目目录、通知文件和机器人处理流程在状态表达上保持一致，避免“同一任务多种状态含义”的问题。

## 适用范围

本规范适用于：

- 项目目录中的 `task.md`
- 机器人目录中的通知文件
- 后续可能增加的结构化元数据文件

## 状态定义

第一版统一使用以下状态值：

### `new`

含义：

- 任务已创建
- 任务内容已具备最小执行信息
- 尚未正式分配给任何机器人

### `assigned`

含义：

- 任务已被分配给某个机器人
- 对应通知文件已生成或准备生成
- 尚未被机器人正式领取执行

### `working`

含义：

- 机器人已接收并开始执行
- 任务可能正在产生中间产物
- 当前不应再被其他机器人重复领取

### `done`

含义：

- 任务目标已完成
- 输出物已回写到任务目录的约定位置
- 任务已具备验收条件

### `failed`

含义：

- 任务执行失败，当前无法自动完成
- 需要人工介入、重新分配或补充输入信息

### `archived`

含义：

- 任务已完成生命周期闭环
- 不再继续修改
- 仅保留查询、审计和复盘用途

## 标准流转路径

标准状态流转如下：

```text
new -> assigned -> working -> done -> archived
                       └------> failed
failed -> assigned
failed -> archived
```

说明如下：

- `new -> assigned`：任务完成准备并被正式分配
- `assigned -> working`：机器人领取任务并开始处理
- `working -> done`：任务完成且结果已回写
- `working -> failed`：处理中发现阻塞或错误，无法继续
- `done -> archived`：任务验收完成后归档
- `failed -> assigned`：补齐信息后重新分配
- `failed -> archived`：确认终止，不再处理

## 非法流转

以下流转默认视为非法：

- `new -> done`
- `new -> working`
- `assigned -> done`
- `done -> working`
- `archived -> working`
- `archived -> assigned`

如果确实需要例外处理，必须在任务说明中记录原因，并由人工确认。

## 单一事实源

任务状态的单一事实源优先级如下：

1. 项目目录中的 `task.md`
2. 后续若引入结构化状态文件，则以结构化文件为准
3. 机器人通知文件中的状态仅作为流程视图，不作为最终事实源

这意味着：

- 任务最终状态必须回写到 `task.md`
- 机器人目录中的状态变化不能代替项目目录中的更新
- 如果项目目录与通知文件状态不一致，以项目目录为准，并视为流程异常

## 各状态下的最小要求

### 当状态为 `new`

任务至少应具备：

- 任务标题
- 背景或上下文
- 目标
- 输入信息
- 验收标准

### 当状态为 `assigned`

除 `new` 的要求外，还应具备：

- 目标机器人
- 分配时间
- 通知文件位置

### 当状态为 `working`

除 `assigned` 的要求外，还应具备：

- 开始执行时间
- 当前执行者
- 执行中备注或中间记录入口

### 当状态为 `done`

除 `working` 的要求外，还应具备：

- 完成时间
- 输出位置
- 完成摘要

### 当状态为 `failed`

除 `working` 的要求外，还应具备：

- 失败时间
- 失败原因
- 是否可重试
- 建议下一步

## 目录动作建议

状态变化应尽量与目录动作保持一致：

- `assigned`：通知文件进入目标机器人 `inbox/`
- `working`：通知文件移动到 `working/`
- `done`：通知文件移动到 `done/`
- `failed`：通知文件移动到 `failed/`

注意：

- 目录动作是工作流表现，不是最终事实源
- 无论通知文件移动到哪里，项目目录中的 `task.md` 都必须同步更新

## 失败与重试规则

对于 `failed` 状态，建议遵循以下规则：

- 先记录失败原因，再决定是否重试
- 同一失败原因不应无限重复重试
- 重新分配前，应先补齐缺失输入或修复阻塞条件
- 如任务已失去业务价值，可直接转为 `archived`

## 人工介入要求

以下情况建议必须人工确认：

- 任务从 `failed` 重新转为 `assigned`
- 任务被判定为 `done` 但缺少明确交付物
- 任务需要跳过某些标准状态
- 任务准备直接终止并归档
