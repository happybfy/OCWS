# 内容注册表规则

本文档定义 OCWS 中项目内容文件的版本追踪与发布管理规则。适用于所有需要通过「内容→网站/输出」流程持续更新的项目。

## 概述

内容注册表（`shared/content-registry.md`）是项目内容文件的**辅助事实源**——它追踪每个内容文件对外发布的状态，解决两类核心问题：

1. **遗漏**：新内容已放入 `shared/input/`，但未创建对应的发布任务
2. **重复**：已发布的内容被再次发布，造成线上重复或覆盖

> 注册表是辅助事实源，不是替代 task.md 的任务分配机制。实际发布仍通过 task.md + 通知文件驱动。

## 状态定义

每个内容文件在注册表中拥有四种状态之一：

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `draft` | 未发布，等待首次发布 | 新文件放入 input/ |
| `published` | 已发布，线上内容与文件一致 | 发布任务完成 |
| `updated` | 已发布过，但源文件被修改 | 人类或机器人修改了 input/ 中的文件 |
| `archived` | 已废弃，不再发布 | 内容下线或不再需要 |

## 状态流转

```
draft ──→ published ──→ updated ──→ published ──→ ...
  │                       │
  └──→ archived           └──→ archived
```

规则：
- `draft` → `published`：首次发布完成
- `published` → `updated`：源文件被修改（时间戳变化或人类标记）
- `updated` → `published`：修改后的内容重新发布完成
- 任意状态 → `archived`：内容下线

## 注册表格式

`shared/content-registry.md` 使用 Markdown 表格：

```markdown
| 文件路径 | 状态 | 首次发布 | 最后更新 | 备注 |
|---------|------|---------|---------|------|
| brand/logo.png | draft | - | 2026-05-29 | |
| news/xxx.md | published | 2026-06-01 | 2026-06-05 | 更新了头图 |
| health/yyy.md | updated | 2026-05-20 | 2026-06-10 | 修改禁忌人群描述 |
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| 文件路径 | ✅ | 相对于 `shared/input/` 的路径 |
| 状态 | ✅ | draft / published / updated / archived |
| 首次发布 | - | 首次发布日期，未发布署 `-` |
| 最后更新 | ✅ | 文件最后修改或注册日期 |
| 备注 | - | 变更原因、注意事项等 |

## 工作流

### 新增内容

1. 人类或上游系统将文件放入 `shared/input/`
2. 将文件路径追加到注册表，标记 `draft`
3. 计划者（Planner）心跳扫描注册表，发现 `draft` 条目
4. 计划者创建发布任务（task.md），将这批 `draft` 条目作为输入
5. 执行者处理完毕后，将注册表中对应条目更新为 `published` + 日期

### 修改已发布内容

1. 人类修改 `shared/input/` 中的文件
2. 将注册表中对应条目状态改为 `updated`
3. 计划者扫描发现 `updated` 条目 → 创建更新任务
4. 执行者完成后 → 状态改回 `published` + 更新日期

### 废弃内容

1. 将注册表状态改为 `archived`
2. 如有对应线上内容需要下线，单独创建任务

## 计划者心跳扫描规则

计划者每次心跳应扫描所有 active 项目的 `shared/content-registry.md`：

1. 收集所有 `draft` 条目 → 按类别（目录）分组，创建内容发布任务
2. 收集所有 `updated` 条目 → 创建内容更新任务
3. 每批次发布/更新任务完成后，验证注册表状态已同步更新

**批量策略：**
- 同类内容可打包为一个任务（如「健康养生文章第 1 批: 20 篇」）
- 单次任务不宜过大：建议每批 10-30 个文件，便于执行和验收
- 余下 `draft` / `updated` 条目等待下批，避免单次任务过于庞大

## 与 task.md 的关系

- task.md 仍然是最权威的事实源
- 注册表是辅助索引，用于快速判断「哪些内容需要发布」
- task.md 的「输入信息」字段应引用注册表路径或具体条目范围
- 冲突时以 task.md 为准

## 初始注册表生成

项目首次启用注册表时：

1. 扫描 `shared/input/` 下所有内容文件
2. 已发布的内容标记为 `published` + 日期
3. 未发布的内容标记为 `draft`
4. 不应存在 `updated` 或 `archived`（初始状态）

## 约束

- ❌ 注册表仅追踪 `shared/input/` 下的内容文件，不追踪代码、配置、模板
- ❌ 注册表不替代 task.md，仅作为计划者的辅助信息
- ✅ 状态变更需同步更新注册表中的 `最后更新` 字段
- ✅ 注册表本身不作为通知分发依据，仅为计划者判断用

## 版本说明

- 当前版本：`v1`
- 生效范围：OCWS 中所有需要内容持续发布的项目
